## 1. 项目简介

本系统是一款专为复杂的数据整理、薪资匹配以及 Excel 并发处理设计的 **数据处理桌面/Web级系统**。它通过构建全栈式的运行环境，将原本容易卡顿或容易人工出错的复杂跨表核算业务彻底自动化，实现了数据秒级读取、自动排版、极速对账与错漏标红等功能。

---

## 2. 技术栈架构

本项目采用 **前后端分离** 的全栈架构：

### 🎨 前端 (Frontend)
- **核心框架**: React 19 (Hooks, Functional Components) + TypeScript
- **构建工具**: Vite 6 (支持极速冷启动和 HMR)
- **UI 组件库**: Tailwind CSS 4, `radix-ui` 组件库设计思想
- **动画设计**: Framer Motion (页面丝滑切换与组件微动画)
- **图标系统**: Lucide React
- **辅助模块**: `concurrently` (支持通过 npm 命令同步启动前后端服务)

### ⚙️ 后端 (Backend)
- **核心框架**: Python 3.10+ & FastAPI (提供高性能异步 API 支持)
- **网络容器**: Uvicorn (ASGI 高性能网络服务器)
- **数据分析层**: pandas, numpy (负责底层大数据块的高效合并、过滤与计算)
- **Excel 持久层**: openpyxl (取代传统慢速 win32com，实现跨平台无缝构建流式文件、单元格渲染与字体加粗等样式固化)
- **并发机制**: `asyncio` & `threading` (应对大表读写的 I/O 阻塞，将进度和日志以流式推送(SSE)至客户端)

## 3. 功能模块说明

### A. 核心薪资核算
在前端**控制台 (Control Panel)**，选择需要跑数据的城市、周期日期或时间段，选中后引擎会自动调度对应的原始数据目录：
- 解析出勤清单，提取所有需要结薪的骑手信息。
- 按城市逻辑整合账单，清洗脏数据（去除空白符、处理长数字等）。
- 把骑手账单按预置规则切分为各大专表及“所得表”。
- **产出结果**：基于模板最终生成标准的总表与独立明细表。

### B. 快速修单：蓝橙单价重跑
在前端右上方的 **动作面板 (Action Panel)** 快速发起：
1. **要求文件**：目标工资表 (`.xlsx`) + 最新的蓝橙/兼职价格档案表 (`.xlsx` / `.csv`)。
2. **执行逻辑**：
   - 读取最新价格表中的打卡人ID和姓名。
   - 解析目标工资表中的【日单量】，替换原有单价，将没有匹配到价格的人员标记为“无单价”。
   - 将“无单价”的连续日期与正常有价格的日期分别智能整合为“备注”语句（如 *1日-3日无单价；4日-7日单价6.*）。
   - 最后，在【配送所得表】中对最新核算的单价与原有薪资方案中的记录进行比对，**自动标红**有差异的异常单元格，防止漏算。

### C. 快速修单：问题单剔除
主要用于客服或风控提供的不合格/异常单号快速从工资中扣除：
1. **要求文件**：目标工资表 (`.xlsx`) + 问题单清单 (`.xlsx`)。
2. **执行逻辑**：
   - 遍历问题单工作簿（包含T10、超时、差评等关键字的 sheet），提取需要剔除的单号。
   - 打开工资表中的【问题单】页，查找对应类型与单号。
   - 自动修改状态为“是” (确认剔除)，并在扣款列中将扣款金额置为 `0` (或依规则自动冲减)。
   - **收尾处理**：引擎会在文件中新建一个红色标签的【剔除源头】子表作为审计归档，并将所有匹配痕迹进行表格宽度自适应与首行加黄色高亮处理。

### D. 兼职已发归档：批量汇总
财务或结算业务的收尾工作（生成汇总大表）：
1. **要求**：选择包含所有兼职历史发放记录报表的本地目录。
2. **执行逻辑**：
   - 扫描该目录下全部 `.xlsx` 工作簿，自动过滤临时文件 (`~$`) 或含有重复字眼的无效表。
   - 将数百个表格中的【配送所得表】及【安全基金】两个 Sheet 下的行列数据抽出。
   - 拼装合并为两个超大宽表，自动剔除空数据、将日期标准化、防止长数字变成科学计数法。
   - 自动添加“工作薄名称”列以便溯源。
   - **完成输出**：生成名为 `<城市>兼职已发汇总.xlsx`，首冻结、内容居中、绘制黑色边框线、智能适配列宽，提供极致的阅读体验。

## 5. 安装与本地部署说明

由于它是全栈系统，分别需要配置 Node.js 和 Python 环境。

### 提前准备
- [Node.js](https://nodejs.org/zh-cn/) (建议 18.x 以上版本)
- [Python](https://www.python.org/downloads/) (建议 3.10 以上版本)

### 步骤一：安装前端依赖
确保你在项目根目录 (包含 `package.json` 的目录)：
```bash
npm install
```

### 步骤二：安装后端依赖 (Python)
进入 `backend` 目录，安装数据分析与服务层依赖：
```bash
cd backend
pip install -r requirements.txt
# 或者使用清华源加速：
# pip install -r requirements.txt -i https://pypi.tuna.tsinghua.edu.cn/simple
```

### 步骤三：一键启动全栈服务
回到项目根目录，使用项目中配置好的并行启动命令运行一切：
```bash
npm run start:local
```
此时终端会打印两束彩色 log：
1. `FRONTEND`: React 的运行地址（如 `http://localhost:3000` 或类似接口）。
2. `PYTHON`: FastAPI 启动地址在 `http://127.0.0.1:8000`。
完成后直接在浏览器打开前端给出的 Local 地址即可使用功能。

## 6. 常见操作流程与注意事项

1. **环境与数据源路径**：前端操作面板会弹出操作系统自带目录选择器（这借助的是后端的 Python 服务唤起），因此**后端必须在本地主线程正确启动，并且不能用无头模式（headless）执行**。
2. **长数字丢失问题**：后端在 `tasks.py` 和 `processor.py` 中利用了一套字符串防吞检测（如遇到 `15位` 甚至 `18位` 的身份证、订单号、银行卡号会在前加入单引号或将其强制类型转为 Python `str` / `@` 文本格式单元格），如果在后期改动了这里的读写逻辑，必须小心科学计数法还原时精度的丢失。
3. **文件占用**：因为 Python 处理 Excel 时会尝试给被处理文件加锁或独占：
   * **运行前**，如遇到目标表在 Office / WPS 中处于打开占用状态时，可能导致保存失败（抛出 `PermissionError`）。此时只要关闭重试即可。
4. **引擎替换升级**：本系统已全量弃用 Windows 底层的 `win32com` 以及 VBS OLE 对象通讯，所有公式比对、颜色标红及列宽拉伸直接操作二进制底层进行 (使用纯血 `openpyxl`)。其优势在于无需等待 Excel 视窗重绘，**性能最高拔高 10 倍以上**，且在 macOS 上不会因缺少组件而崩溃。

## 7. 后期开发与维护指南

### UI调整与页面添加
前端采用 Tailwind CSS 4，你可以打开 `src/components/RightPanel.tsx` 或类似大面板修改，样式遵循纯函数式思想，无需顾虑类名冲突。

### 核心公式分析引擎 (`processor.py`)
如果 HR / 财务变更了未来的业务排版方案（如：要求新增一列计件，或薪给制变更），你需要修改：
- `backend/processor.py` 中寻找拼装字典列（查找关键词：`df_target` 或是各种表头的映射数组与 `pd.DataFrame` 结构列名）。

### 持续通信拦截 (`tasks.py` 里的 `yield`)
本项目利用 `StreamingResponse` 返回类似长链接的文本推流：
```python
yield create_progress_event(0.5)
yield create_log_event("进度通知", "INFO")
```
未来在增加其它耗时数分钟的任务时，**必须**遵循这一生成器模式 `_gen()`，在长时间循环的代码块塞入 `asyncio.sleep(0.01)` 以便避免主线程阻塞并及时向 UI 组件推送 `Progress` 回调，这样前端的日志面板才能像机器一样酷炫的实时打字反馈。
